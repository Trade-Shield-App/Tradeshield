// The Review Problem — Solved

REVIEWS THAT
MEAN SOMETHING.

The UK trades industry is full of fake reviews — traders buying them, clients leaving revenge ratings mid-job. TradeShield has made both impossible at the database level.

📋
Job posted & bid accepted
Review system is locked. Neither party can submit. Not in the UI. Not via the API.
🔒
Escrow locked — work begins
Payment secured by Stripe. Reviews locked throughout the entire job.
🔐
Trader marks complete
Client notified to inspect work. Reviews still locked — job not approved yet.
✅
Client approves — payment released
Only at this moment does the database unlock reviews for both parties.
⭐
Both leave honest reviews
Neither side can read the other's review before writing their own. Both go live together.
HOW THE LOCK WORKS

This is not a UI restriction. It is a PostgreSQL BEFORE INSERT trigger on the reviews table itself.

🗄️
Every attempt to write a review row fires a BEFORE INSERT trigger.
🔍
It checks: is jobs.status = 'completed'? Are client_can_review and trader_can_review both TRUE?
🚫
If either check fails, the INSERT is rejected with a hard exception. The row never exists.
🔑
Those flags are only set by complete_job() — a server-side function called exclusively when the client approves payment release.
👁️
Reviews are submitted privately and go live only when both submit — or 14 days pass. Neither reads the other's review before writing their own.
🔐No UI workaround. No API bypass. The hardest review integrity guarantee on any UK trades platform.
